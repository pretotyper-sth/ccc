# 일본 전환율·트래킹 분석 및 대안

CONTEXT_HANDOFF.md, CHAT_ANALYTICS_EVENTS.md 기준.  
**일본인 사고방식**, **트래킹/폼 오류 가능성**, **여행 제약(한국 와야만 참여)** 까지 한 번에 점검·설계하기 위한 문서.

---

## 1. 일본인 관점에서 전환율이 낮았을 때 의심할 요인

### 1.1 심리·사고방식

| 요인 | 설명 | 대안 |
|------|------|------|
| **즉시 참여 부담** | "참여할 계획 있어요"를 누르면 곧바로 약속한 것 같은 느낌. 일본에서는 **曖昧(あいまい)한 단계**를 선호. | 선택지를 "참여를 검토 중", "정보만 먼저 받고 싶다"처럼 부담을 낮추거나, "나중에 참여할 때 연락받기" 옵션을 명시. |
| **개인정보·동의 민감도** | 이메일·성별·연령 등을 한 번에 주는 것이 부담. **단계적 노출**을 선호. | 챗 단계는 유지하되, 완료 문구에서 "개인정보는 매칭·연락 목적으로만 사용"을 짧게 한 줄 더 명시. |
| **실제로 참여 가능 시점이 다름** | 한국 유저는 "지금 서울에 있음" / 일본 유저는 "언제 한국 갈지 미정". **when_join**이 "미정"이어도 "그때 가면 연락해 주세요"가 없으면 기억에서 사라짐. | **여행 예정 시 알림받기** 플로우 추가(아래 4장). |
| **불안·신뢰** | "통역 있는 3:3"이 구체적으로 어떻게 진행되는지, 누가 오는지 모르면 **참여를 미룸**. | 후기 3개 옆에 "운영진 소개" 또는 "첫 세션 무료/쉬운 설명" 한 줄 추가 검토. |
| **언어 전환 의심** | `/jp`가 진짜 일본어 전용인지, 한국 측에 데이터가 넘어가는 건 아닌지 불안. | 푸터나 완료 화면에 "응답は韓国側の運営のみが確認します" 등 **데이터 사용 주체**를 한 줄 명시. |

### 1.2 UX·플로우

- **진행 단계가 12단계**로 길게 느껴질 수 있음. 일본 사용자는 **중간 이탈** 시 "다시 처음부터"가 부담.
- **location**이 "강남, 성수, 한남, 홍대"로 고정되어 있으면, "아직 한국 가는 일정이 없어서 고를 수 없다"고 느낄 수 있음.
- **when_join**에 "미정 (정보 받으면 그때 결정)"이 있어도, "미정인 사람에게 나중에 연락"하는 장치가 없으면 전환으로 이어지지 않음.

## 2. 트래킹·폼 제출이 안 되는 경우 점검 (한국은 되는데 일본만 0건일 때)

### 2.1 가능한 원인

| 구분 | 가능성 | 점검 방법 |
|------|--------|-----------|
| **GA4 / 광고 픽셀** | 일본 트래픽만 필터링·블록되는 경우 (리전, 브라우저 확장) | GA4에서 `page_lang=jp` 이벤트가 아예 0인지 확인. `chat_funnel_start` 기준으로 JP 진입 수 확인. |
| **Formspree** | 같은 Formspree 폼 ID를 쓰므로 **기술적으로 KR/JP 구분은 `_source` 필드**로만 가능. JP에서만 CORS/네트워크 오류가 나는 경우는 드묾. | Formspree 대시보드에서 `_source: chat_jp` 로 필터해 제출 건수 확인. 0건이면 **제출 자체가 안 된 것**. |
| **클라이언트 오류** | 이메일 검증 실패, `res.ok` 실패 시 사용자에게만 에러 표시되고 서버에는 안 남음. | GA4 `chat_submit_error` (type: email_required / email_invalid / server / network) 를 **page_lang=jp** 로 집계해 JP에서 에러가 많은지 확인. |
| **진입 자체가 적음** | 일본 유저가 `/jp`에 거의 안 들어옴. | `chat_funnel_start` 중 `page_lang=jp` 비율 확인. |

### 2.2 권장: 정기 점검 체크리스트

다음 항목을 **주 1회 또는 캠페인 후** 확인하면 "한국은 되는데 일본만 0건" 원인을 구분할 수 있다.

1. **GA4**
   - `chat_funnel_start` 건수: `page_lang=kr` vs `page_lang=jp`
   - `chat_submit_click` vs `chat_submit_success`: JP에서 click은 있는데 success가 0이면 → **폼 제출 실패(네트워크/서버/검증)** 가능성
   - `chat_submit_error`: `page_lang=jp` 일 때 `type` 별 건수 (email_required, email_invalid, server, network)

2. **Formspree**
   - 제출 건수를 **필드 `_source`** 로 필터:
     - `_source = chat_kr` → 한국 챗 퍼널
     - `_source = chat_jp` → 일본 챗 퍼널
   - JP 0건인데 GA4에서는 `chat_submit_success` 가 JP에서 1건이라도 있다면 → Formspree 필터/설정 이슈 가능성(드묾).

3. **코드 레벨**
   - `ChatPage.tsx` 는 KR/JP 동일 컴포넌트. `copy.formSource` 만 다름. 따라서 **같은 코드 경로**로 제출됨.
   - Formspree URL이 KR/JP 동일하므로, **제출 실패 시 원인은 클라이언트(검증/네트워크) 또는 Formspree 서버**로 한정.

### 2.3 코드에서 해둘 일 (진단 강화)

- 모든 퍼널 이벤트에 이미 `page_lang` 이 붙어 있음 (`lib/analytics.ts`: `path.startsWith("/jp") ? "jp" : "kr"`).  
- **추가 권장:** `chat_submit_click` / `chat_submit_success` / `chat_submit_error` 에 `form_source: copy.formSource` 같은 파라미터를 하나 더 보내면, GA4에서 "chat_jp" 기준으로만 필터해도 되어 **JP 0건 여부**를 더 쉽게 확인 가능.

## 3. 일본인만의 제약: "한국에 와야만 참여 가능"

### 3.1 문제 정리

- **한국 유저**: 서울(또는 현지)에 있으므로 "참여 시기"만 정하면 됨.
- **일본 유저**: 대부분 **일본 거주**. 참여하려면 **한국 여행이 선행**되어야 함.  
  → "지금 당장 참여할 계획 없음" + "나중에 한국 갈 때 연락받고 싶음" 수요가 있을 수 있음.

### 3.2 방향

- **옵션 A (챗 퍼널 내)**  
  - `when_join` 또는 purpose 단계에  
    - "**한국에 갈 예정일은 아직 없음. 일정이 정해지면 연락해 주세요**" 같은 선택지 추가.  
  - 선택 시 → 이메일만 받고, "여행 예정 시 알림" 용도로 저장. 기존 Formspree에 `_remind_when_travel: 1` 같은 필드로 보내도 됨.

- **옵션 B (별도 경로)**  
  - `/jp` 또는 GNB에 **「韓国に行く頃に知らせてほしい」** 링크.  
  - 최소 폼: 이메일 + "예정 시기(예: 3개월 이내 / 6개월 이내 / 미정)".  
  - Formspree에 `_source: notify_when_travel_jp` 로 제출 → 나중에 메일 발송 시 이 리스트만 따로 타게팅.

- **옵션 C (완료 화면)**  
  - 챗 완료 후 "일본에서 오시는 분은, 한국 여행 일정이 잡히면 알려주시면 그에 맞춰 세션을 안내해 드립니다" 문구 + (선택) "여행 일정 알림 신청" 링크.

위를 조합해도 됨(예: 챗에서 "미정·나중에 연락" 선택 시 → 완료 화면에서 "여행 알림" CTA 한 번 더 노출).
